# LP Motion v2 — hero scroll-scrub — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Le hero vidéo des LP devient un scrollytelling (frames scrubbées au scroll), avec extraction automatique des frames par un worker ffmpeg sur le bot Railway.

**Architecture:** État `hero.frames` dans lp_config (jsonb, idempotent par sourceUrl). Worker à poll côté bot (pattern campaign/status) : télécharge la vidéo, extrait ~90 frames WebP via ffmpeg, les uploade dans lp-media, marque ready/failed. Côté LP, un composant client HeroScrub (canvas + ScrollTrigger, section épinglée) remplace le hero quand les frames sont prêtes — fallback intégral sinon.

**Tech Stack:** ffmpeg (Dockerfile bot), Node spawn, Supabase Storage service-role, Next.js LP existante, GSAP ScrollTrigger + Lenis (déjà présents), Vitest.

## Global Constraints

- Aucune migration SQL, aucune nouvelle policy storage (worker en service role ; lecture publique par URL).
- Idempotence stricte : extraction seulement si `mediaType==='video'` ET (`frames` absent OU `frames.sourceUrl !== hero.mediaUrl`). `failed` sur le même sourceUrl = skip (pas de boucle de retry).
- Paramètres d'extraction : `fps=6`, `scale=960:-2`, WebP `-quality 70`, frames nommées `f-0001.webp`… (padding 4), chemin `${restaurantId}/frames/${hash}/`. hash = 12 premiers hex de sha256(sourceUrl).
- Worker : poll `LP_FRAMES_POLL_MS` (défaut 60000), démarrage loggué `[lpframes-worker] démarré`, aucune exception ne sort de la boucle.
- LP : zéro régression pour heros image / frames absentes / pending / failed (fallback = rendu actuel). `prefers-reduced-motion` → première frame statique, pas de pin.
- Textes FR, tokens/design intacts, pas de nouvelle dépendance web ; bot : aucune dépendance npm nouvelle (spawn ffmpeg système).
- Gate par tâche : typecheck + tests du paquet touché ; suites existantes vertes (web 42, bot 51). Jamais de `next build` pendant qu'une preview tourne.
- Branche : `feature/lp-motion-v2`.

**Interfaces:**
- Consumes: helpers Task 2 ; client Supabase service role du bot (même construction que statuses/repo.ts).
- Produces:
  - Repo : `listCandidates(): Promise<{ restaurantId: string; mediaUrl: string; lpConfig: unknown }[]>` (select id, lp_config des restaurants, filtrage needsExtraction côté JS) ; `setFrames(restaurantId, frames)` (merge lp_config.hero.frames via update service role — relire le lp_config courant juste avant l'update pour ne pas écraser les éditions admin) ; `uploadFrame(path, buffer)` (storage lp-media, contentType image/webp).
  - `processOne(candidate, deps)` : pending → download (fetch, buffer → fichier temp via node:fs) → spawn ffmpeg (promisifié, stderr capturé) → lecture des frames produites (tri) → upload séquentiel → probe dimensions de la 1re frame (parse stderr ffmpeg `960x540` OU lire via ffprobe ; plus simple : largeur 960 fixe + hauteur calculée depuis le ratio ffprobe — si indisponible, stocker 0×0, la LP n'en dépend que pour le ratio initial et sait retomber sur 16/9) → setFrames ready. Catch → setFrames failed (message tronqué en log).
  - `startLpFramesWorker({ repo, pollMs, runFfmpeg })` : poll try/catch/finally, log démarrage.
- Tests : worker avec deps mockées (candidat vidéo → pending puis ready avec count correct ; échec ffmpeg → failed sans throw ; aucun candidat → no-op). Pas de vrai ffmpeg dans les tests.
